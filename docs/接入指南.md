# Lurus Platform 接入指南（中文）

> 给团队成员/朋友看的接入说明。如果只想"我新做了一个项目，要让用户能登录注册",直接看本文。
> 老的英文 SDK 详细参考见 `integration-guide.md`。

最后更新：2026-04-28（Phase 4 ops 抽象 + Phase 5 Sprint 1 API key 抽象上线后）

---

## 0. 一句话理解

Lurus 平台 = **统一身份 + 钱包 + 计费 + 通知**。
所有产品（Tally / Lucrum / Memorus / Forge / Switch / 路途 APP …）共用同一套账号体系，**不要自己造账号系统**。

---

## 1. 三种使用姿势（按场景选）

| 场景 | 你需要的东西 | 走的路 |
|------|------------|------|
| **A. 用户在你的网页/APP 登录注册** | OIDC 客户端（自动） | Zitadel OIDC 标准流程 |
| **B. 你的后端要查/扣用户余额、查权限、发起退款** | INTERNAL_API_KEY | `/internal/v1/*` REST |
| **C. 你的服务要订阅业务事件**（用户充值/订阅到期等） | NATS 凭证 | NATS JetStream |

绝大多数项目同时用 **A + B**。C 是优化项（缓存失效用）。

## 2. 场景 A：用户登录注册（OIDC）

### 2.1 用户体验

用户访问 `your-product.lurus.cn` → 点"登录" → 跳转 `auth.lurus.cn`（Zitadel 的 Lurus 登录页）→ 用户用以下任一方式登录：

- ✅ **手机号 + 短信验证码**（已上线，阿里云短信）
- ✅ **邮箱 + 密码**
- ✅ **微信扫码**（已上线，OAuth2 适配器）
- ✅ **路途 APP 扫码登录**（已上线，QR primitive v2）
- ⚠️ Apple / Google / GitHub 登录：Zitadel 支持，需要时再开

注册和登录是**同一入口**——新用户首次输入手机号时会走注册校验流程，不需要分开按钮。

### 2.2 你（接入方）要做的

#### 步骤 1：在平台仓库给你的应用加一行

编辑 `2l-svc-platform/config/apps.yaml`，加：

```yaml
apps:
  # 你的项目
  - name: your-product
    display_name: Your Product
    environments:
      - env: prod
        domain: your-product.lurus.cn
        secret:
          namespace: your-namespace      # K8s namespace
          name: your-product-secrets     # K8s Secret 名
          client_id_key: ZITADEL_CLIENT_ID
          client_secret_key: ZITADEL_CLIENT_SECRET  # 仅 confidential client 需要
        restart_deployment: your-product  # K8s deployment 名（写完 secret 自动重启）
    oidc:
      app_type: web
      auth_method: basic        # 后端项目 = basic（机密）；纯前端 SPA = none（PKCE）
      grant_types: [authorization_code, refresh_token]
      response_types: [code]
      redirect_path: /api/auth/callback/zitadel  # 你后端的回调路径
      post_logout_path: /
```

提交 PR → merge → ArgoCD 同步后，平台的 **app_registry reconciler** 会自动：

1. 在 Zitadel 创建/校准 OIDC client（grant_types、redirect_uris、auth_method 全部按 yaml）
2. 把 `client_id` 和 `client_secret` 写进你的 K8s Secret
3. 重启你的 deployment 让它读到新 secret

**你完全不需要登录 Zitadel 控制台。** 删除条目也安全——reconciler 不会自动删 Zitadel 的 app（破坏性操作走 Phase 4 的 QR-delegate + 路途 APP 生物识别）。

#### 步骤 2：你的后端读 secret 并验证 JWT

环境变量（K8s Secret 里已经有）：

```bash
ZITADEL_ISSUER=https://auth.lurus.cn
ZITADEL_JWKS_URL=https://auth.lurus.cn/oauth/v2/keys
ZITADEL_CLIENT_ID=...     # reconciler 自动写
ZITADEL_CLIENT_SECRET=... # reconciler 自动写（仅 basic auth_method）
```

后端代码（Go 示例）：

```go
import "github.com/coreos/go-oidc/v3/oidc"

provider, _ := oidc.NewProvider(ctx, os.Getenv("ZITADEL_ISSUER"))
verifier := provider.Verifier(&oidc.Config{ClientID: os.Getenv("ZITADEL_CLIENT_ID")})

// 用户带 Bearer JWT 来 → 校验
idToken, err := verifier.Verify(ctx, rawJWT)
var claims struct {
    Sub   string `json:"sub"`
    Email string `json:"email"`
}
idToken.Claims(&claims)
// claims.Sub 就是用户在 Zitadel 的 user_id
```

#### 步骤 3：把 Zitadel 用户映射到 Lurus account

第一次见到这个用户时，调平台的 internal API 把他注册进 Lurus 的 account 表：

```bash
POST http://platform-core.lurus-platform.svc:18104/internal/v1/accounts/upsert
Authorization: Bearer ${INTERNAL_API_KEY}
Content-Type: application/json

{
  "zitadel_sub": "370559694772633711",
  "email": "user@example.com",
  "username": "alice"
}
```

返回 `{ account_id, lurus_id, ... }`。**之后用 `account_id` 而不是 `zitadel_sub` 做业务关联**——Lurus 的 account_id 是 int64 主键，更稳定。

⚠️ Lurus 平台的"用户"和 Zitadel 的"用户"是**两层**：
- Zitadel 管认证（密码/MFA/OAuth）
- Lurus 管业务（钱包/订阅/VIP/积分）
- 两者通过 `zitadel_sub` 关联

## 3. 场景 B：服务到服务调用（要 INTERNAL_API_KEY）

### 3.1 怎么拿 key

**Phase 5 Sprint 1 之后（已上线）**：

1. 打开 `https://identity.lurus.cn/admin/`（Lurus 管理面板，需要管理员登录）
2. 点 **"应用密钥"** Tab
3. 点 **"+ 新建密钥"**
4. 填写：
   - **名称**（必填）：`your-product-prod`，3-64 字符，小写字母/数字/dash/underscore，**这个就是幂等键**——同名第二次创建直接返回已有项不会换发
   - **显示名**：`Your Product (Production)`
   - **用途**：选 `外部集成` 或 `MCP 服务` 等
   - **过期天数**：默认 365，0 = 永不过期。**生产建议 ≤ 365 天**
5. 点"创建" → **token 只显示一次**，立刻复制保存
6. 把 token 填进你的 K8s Secret：`INTERNAL_API_KEY=<token>`

底层做了什么（你不需要知道，但写在这里方便排错）：
- 平台调 Zitadel 创建一个 Service User（machine user）
- 给它发一个 PAT（Personal Access Token）
- token 的 SHA256 存在 Lurus 的 `identity.api_keys` 表里（仅审计指针）
- token 原文不落库，**丢了只能换发**

### 3.2 怎么用 key

```go
req, _ := http.NewRequest("GET", baseURL + "/internal/v1/accounts/by-id/" + id, nil)
req.Header.Set("Authorization", "Bearer " + os.Getenv("INTERNAL_API_KEY"))
resp, _ := http.Do(req)
```

或者用平台提供的 SDK（Go）：`pkg/platformclient/`，详见英文版 `integration-guide.md`。

### 3.3 主要 internal API 一览

| 端点 | 用途 |
|------|------|
| `GET /internal/v1/accounts/by-zitadel-sub/:sub` | JWT 验证后查到 sub，再查 account |
| `GET /internal/v1/accounts/by-email/:email` | 按邮箱查 |
| `GET /internal/v1/accounts/by-phone/:phone` | 按手机号查 |
| `POST /internal/v1/accounts/upsert` | 首次见到 Zitadel 用户时同步 |
| `GET /internal/v1/accounts/:id/entitlements/:product_id` | 查权限/套餐（缓存 5 分钟） |
| `GET /internal/v1/accounts/:id/wallet/balance` | 查余额 |
| `POST /internal/v1/accounts/:id/wallet/debit` | 扣款（小额即时） |
| `POST /internal/v1/accounts/:id/wallet/pre-authorize` | 冻结估算（流式 API 用） |
| `POST /internal/v1/wallet/pre-auth/:id/settle` | 按实际成本结算 |
| `POST /internal/v1/wallet/pre-auth/:id/release` | 释放冻结（请求失败时） |
| `POST /internal/v1/checkout/create` | 余额不足时跳支付 |
| `GET /internal/v1/checkout/:order_no/status` | 轮询订单状态 |
| `POST /internal/v1/sms/relay` | （平台内部用）Zitadel SMS webhook 转阿里云 |

### 3.4 token 换发 / 撤销

- **换发**（旧 token 立刻失效，发新 token）：在管理面板对应行点"换发"。脚本调 `POST /admin/v1/api-keys/:name/rotate`。
- **撤销**（永久作废，不可恢复）：点"撤销"。脚本调 `DELETE /admin/v1/api-keys/:name`。
- 撤销后**同名重建会保留历史审计**（reincarnate 模式：新行复用 id，状态从 revoked → creating → active）。

## 4. 场景 C：订阅业务事件（NATS）

```go
nc, _ := nats.Connect(os.Getenv("NATS_ADDR"))  // nats://nats.messaging.svc:4222
js, _ := nc.JetStream()

js.QueueSubscribe("identity.entitlement.updated", "your-product-consumer", func(msg *nats.Msg) {
    var ev struct {
        AccountID int64  `json:"account_id"`
        ProductID string `json:"product_id"`
    }
    json.Unmarshal(msg.Data, &ev)
    if ev.ProductID == "your-product" {
        cache.Delete(ev.AccountID)
    }
    msg.Ack()
})
```

主要 subjects：

| Subject | 触发时机 |
|---------|---------|
| `identity.account.created` | 新用户注册 |
| `identity.subscription.activated` | 套餐开通 |
| `identity.subscription.expired` | 套餐到期 |
| `identity.topup.completed` | 充值到账 |
| `identity.entitlement.updated` | 权限变更 |
| `identity.vip.level_changed` | VIP 升降级 |

## 6. 接入清单（朋友可以照着 check）

```
□ 1. 在 2l-svc-platform/config/apps.yaml 加你的应用条目，提 PR
□ 2. PR merge 后等 5 分钟 → kubectl get secret -n <你的 namespace> 看到 ZITADEL_CLIENT_ID 已写入
□ 3. 后端用 ZITADEL_ISSUER + ZITADEL_JWKS_URL 校验 JWT
□ 4. 第一次见到用户 → POST /internal/v1/accounts/upsert（幂等）
□ 5. 之后所有业务关联用 Lurus account_id（int64）
□ 6. 需要查权限 → GET /internal/v1/accounts/:id/entitlements/:product_id（自带 5min 缓存）
□ 7. 需要扣费 → 小额用 debit；大额或流式用 pre-authorize → settle/release
□ 8. 余额不足 → POST /internal/v1/checkout/create 拿支付 URL
□ 9. internal API key 通过 https://identity.lurus.cn/admin → "应用密钥" Tab 自助生成
□ 10. 监控接入：你的服务 emit qr_delegate_confirms_total / 业务 metric → 平台 Grafana 已统一抓取
```

## 7. 排错速查

| 症状 | 可能原因 | 怎么查 |
|------|---------|--------|
| 跳到 auth.lurus.cn 后报 `client not found` | apps.yaml 还没 merge / reconciler 没跑 | `kubectl logs -n lurus-platform deploy/platform-core | grep app_registry` |
| 拿到 JWT 但 `/accounts/by-zitadel-sub/:sub` 返 404 | 用户没 upsert 过 | 调 `/internal/v1/accounts/upsert` 同步进来 |
| internal API 返 401 | INTERNAL_API_KEY 错 / 过期 / 撤销 | 管理面板看密钥状态；过期就换发；撤销就重建 |
| 充值成功但权限没更新 | 缓存没失效 | 订阅 NATS `identity.entitlement.updated` 主动 invalidate |
| 用户在 OIDC 改密码后旧 JWT 还能用 | JWT 默认 24h，OIDC 不会强制踢人 | 改密码后用 `revoke session` 端点（计划中）或等 JWT 自然过期 |
| 短信发不出 | 阿里云模板审核未过 / 签名不对 | `kubectl logs deploy/platform-core | grep sms_relay` 看实际错误码 |
