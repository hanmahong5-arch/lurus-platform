# Lurus Platform 接入指南（简化版）

> **承诺**：你的产品接入用户登录 = **2 行代码**。不需要 PR，不需要 K8s，不需要 client_id/secret，不需要懂 OIDC/JWT/JWKS。
>
> 多租户、计费、订阅 也都隐藏在 `account_id` 后面，产品方默认看不到。

最后更新：2026-04-28（简化版 v2）

---

## TL;DR — 你的产品要做的全部事情

### 如果你的域名是 `*.lurus.cn`（推荐，零配置）

**前端**（HTML 里加一行）：
```html
<script src="https://cdn.lurus.cn/lurus-auth.js"></script>
```

然后随时调：
```js
await lurus.login()              // 弹出登录页
const user = await lurus.whoami() // {account_id, email, plan}
await lurus.logout()
```

**后端**（任何语言）：
```bash
# 用户带着浏览器 cookie 来 → 你转发给 platform 问"这个用户是谁"
curl https://identity.lurus.cn/api/v1/whoami \
  -H "Cookie: $forwarded_user_cookie"
# → {"account_id": 12345, "email": "...", "plan": "pro"}
```

完。**没有 client_id，没有 secret，没有 K8s，没有 PR。**

### 如果你的域名是外部（不是 *.lurus.cn）

打开 `https://identity.lurus.cn/admin/` → "应用密钥" Tab → "+ 新建密钥" 填一下表 → 拿到 `client_id` 和 `client_secret`。后端走标准 OIDC 验证。

仍然不需要 PR，也不需要懂 K8s。

---

## 多租户（你不需要懂）

Lurus 内部 = `account` + `org` + `org_member` 三层。
**对你的产品而言，只有 `account_id`**。

- 老板登录 → 拿到 `account_id=1`
- 员工登录 → 拿到 `account_id=2`
- 他们之间通过 `org_id` 关联，**但你的业务表只用 `account_id` 做外键**
- 你的产品要不要支持组织功能？要的话调 `GET /api/v1/account/:id/orgs` —— 返回这人在哪些组织里，每个组织里他是什么角色。**不要的话忽略它**。

> 99% 的工具型产品（Tally / Lucrum / Memorus / 路途 ...）只用 account_id 就够了。组织视图是产品自己的业务决策，不是平台逼你做。

## 计费/订阅（你也不需要懂）

```bash
# 用户买了什么套餐？能用什么权限？
GET https://identity.lurus.cn/internal/v1/accounts/:id/entitlements/:product_id
# → {"plan_code": "pro", "max_requests": "10000", "expires_at": "..."}

# 余额够不够？要扣钱了
POST https://identity.lurus.cn/internal/v1/accounts/:id/wallet/debit
{"amount": 0.5, "reason": "api_call", "product_id": "your-product"}
# 余额不够 → 返回 ErrInsufficientBalance → 跳支付页
```

更详细的计费 pattern（预授权 / 流式 / 退款 / 充值跳转）见英文版 `integration-guide.md`。

## 接入方需要"准备"的东西

| 你想做的 | 你要准备的 |
|---------|-----------|
| 用户登录（基础） | **0**。装 SDK。 |
| 后端查用户身份 | **0**。转发 cookie 给 `/api/v1/whoami`。 |
| 后端调计费/查权限 | **1 个 token**：在 `identity.lurus.cn/admin/` 自助生成（"应用密钥"Tab，Phase 5 Sprint 1 已上线）。 |
| 订阅事件（可选） | **NATS 凭证**：找平台管理员要。仅在你需要主动失效缓存时才需要。 |

**你不需要准备**：
- ❌ Zitadel client_id / secret（`*.lurus.cn` 子域 = 零，外部域 = 一键自助生成）
- ❌ K8s manifest / PR / RBAC / Secret 编排
- ❌ 自己实现 JWT 验证 / JWKS 拉取 / 签名校验
- ❌ 数据库里建 `user` 表（用 Lurus 的 `account_id` 当外键即可）
- ❌ 短信通道 / 邮件通道（登录注册流走平台的，你不碰）

## 现在的可靠性（诚实评估）

按"你接入是否能跑通"分档：

### 已经可用（生产）✅

| 能力 | 状态 |
|------|------|
| `*.lurus.cn` 子域 + Zitadel 标准 OIDC 登录（带 client_id/secret） | ✅ 跑了一年 |
| `/internal/v1/whoami` 类接口（实际是 `/account/me` + `/accounts/by-id` + `/accounts/validate-session`） | ✅ 现役 |
| `/admin/v1/api-keys` 自助生成 INTERNAL_API_KEY | ✅ 2026-04-28 刚上线 |
| 多租户（org / org_member / org_api_key） | ✅ Lucrum 在用 |
| 钱包扣款 / 预授权 / 订阅 / 多支付通道 | ✅ |

### 还需要补一刀才到"两行接入"承诺 ⚠️

下面这些是**简化版 SDK 的最后一公里**，目前还需要 1-2 天集中工作：

| 缺口 | 改动量 | 怎么解 |
|------|-------|-------|
| `/api/v1/whoami` 公开端点（接 cookie 而非 JWT） | ~30 行 Go | 包一层 `/account/me` + 自动从 `Cookie: lurus_session=...` 解析 |
| Cookie 配 `Domain=.lurus.cn` 让子域共享 | 改 1 个配置 | 现在登录后 cookie 域名是 `auth.lurus.cn`，子域读不到 |
| `cdn.lurus.cn/lurus-auth.js` SDK | ~80 行 JS | 简单包一下 OAuth popup + cookie 读取 |
| 自有域产品的"一键 onboard" Web 表单 | ~150 行（前端 + 后端） | 现在 apps.yaml 是 PR；改成 Web 表单 → 后端写 yaml 的封装 |

**结论**：底层全有了，门面还没贴。下次集中改造把这 4 项 ship 掉，那时这份文档的"承诺"就是 100% 兑现的。

## 现在该怎么办（朋友的实际操作步骤）

### 路径 A：朋友想"立刻接入个登录看看"

1. 域名用 `*.lurus.cn` 子域（找 Lurus 管理员要一个，例如 `friend.lurus.cn`）
2. 现阶段（2026-04-28）：暂时仍走标准 OIDC 流程
   - 在 platform 仓库 `config/apps.yaml` 加一个块（10 行 yaml，例子在仓库里翻）
   - PR merge → 5 分钟内 K8s secret 自动写入 → 后端读 `ZITADEL_CLIENT_ID/SECRET`
   - 后端用 `golang.org/x/oauth2` 或 `next-auth` 之类标准库即可
3. 等"两行接入"上线后（下个迭代），删掉这一切，换成 SDK 一行

### 路径 B：朋友想"看清楚不踩坑"

让他翻两份东西就够了：
- 这份 `接入指南.md`（你正在读的）— 看认知模型
- `docs/integration-guide.md`（英文）— SDK 详细 API

不要再翻别的 Zitadel/K8s/RBAC 文档。**那些是平台运维的事，不是接入方的事。**

## 排错（仅供平台运维参考，朋友不需要看）

| 症状 | 通常原因 |
|------|---------|
| 跳转 auth.lurus.cn 后报 `client not found` | apps.yaml 还没 merge |
| `/whoami` 返 401 | cookie 没正确转发 / cookie 域不对 |
| `/internal/v1/...` 返 401 | INTERNAL_API_KEY 被换发 / 撤销 |
| 充值成功但权限没更新 | 缓存未失效 — 主动调 entitlement 接口或订 NATS |
