## 评估维度

每条条目需通过 4 道关：

1. **可靠**：所有上游/下游故障模式都被显式处理（NewAPI down / Redis down / DB down / NATS 重投 / 时钟漂移）
2. **安全**：默认拒绝；密钥不进日志；端点鉴权 + 速率限制；CORS/CSP/Cookie 属性正确
3. **成熟**：有运维 runbook；schema 变更可回滚；CHANGELOG；版本化契约；不破坏接入方
4. **可观测**：每个关键操作出 metric + structured log；/readyz 反映依赖；trace span 跨服务

完成验收：
1. `OnTopupCompleted` 接 `eventID` 参数；空 ID → 处理但记 warning

3. Redis SETNX miss → 处理；处理失败 → NewAPI 错误传播 NAK，dedup key 自动 TTL 过期允许重试
4. Redis call error → fail-CLOSED（返错让 JetStream NAK + 退避重试，不双扣）
5. 三个 newapi_sync 操作（account_created / topup / llm_token）出 prometheus counter，labels = `op,result`

## 决策原则（写在硬化清单里防漂移）

- **钱相关 = fail-closed**。出错宁可暂停同步堆积重试，也不允许"反正我也不知道，先扣了再说"
- **接入面任何错误都要带机器可读 code**。`{"error_code": "newapi_sync_disabled", "message": "..."}` 让客户端可分支
- **新加 metric 必须同时加 alert rule**（暂存到 `docs/observability/alerts.md`，下次集中接入 Alertmanager）
- **任何对外行为变更都要更新 CHANGELOG**（P1-8 落地后开始）
