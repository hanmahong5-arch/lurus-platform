最后更新：2026-04-29

---

## 长期愿景（North Star）

**一行接入，十万租户**：

任何 `*.lurus.cn` 子域产品，只要在 HTML 里加一行：
```html
<script src="https://cdn.lurus.cn/lurus-auth.js"></script>
```
即获得：登录、注册、统一身份、计费查询、权限校验，全部能力。

后端只需调一次：
```
GET https://identity.lurus.cn/api/v1/whoami
```
取回 `{account_id, plan}`。够了。

不需要：OIDC client_id / JWKS / Zitadel project / apps.yaml PR / K8s Secret 编排 / NextAuth 配置 / multi-tenant org 管理。

## 关键原则（这些不能违反）

1. **不要再引入"必填的配置"**。每加一个 env var，就是给接入方加一份认知负担。
2. **不要在 platform 之外要求改动**才能"简化"接入。如果 X 产品要改 50 行代码才能享受新模型，模型就还不够好。
3. **Dogfood 优先**。一个真实产品跑通新路径 > 十份完美的设计文档。
4. **不怕多次迭代**。drop-in 模式自己也会演进；先 ship 能 work 的最小集，再补齐细节。
5. **删 > 加**。任何"简化方案"如果新增了文件/配置/CI 步骤，重做。

## 实施序列

按"长期收益密度"排序。每一步都是**单独可交付**的，不依赖未完成的下一步。

### 阶段 1 — 最小可行 drop-in（解锁第一个产品）

| # | 工作 | 大小 | 价值 | 状态 |
|---|------|------|------|------|
| 1 | `GET /api/v1/whoami` 端点（吃 cookie 或 Bearer） | ~30 行 Go + test | **支撑 drop-in 的核心接口** | ✅ 2026-04-29 ship (`6545034`) |
| 2 | 登录端点种 `lurus_session` cookie 到 `.lurus.cn` 父域 | ~10 行 Go | 让子域共享会话 | ✅ 2026-04-29 (cookie attrs: Domain/Secure/HttpOnly/SameSite=Lax 全到位) |

| 4 | Lucrum 切到 drop-in（删 NextAuth/OIDC，调 /whoami + LLM 走 NewAPI） | ~80 行 TS + NewAPI 同步链 | **第一个 dogfood，证伪/证实模型** | 🟡 拆成 4a/4b/4c/4d，见下方 NewAPI 同步序列 |

## 跨阶段：必须长期维护的事

- **接入指南随每次破坏性变更同步更新**。文档落后于代码 = 接入方踩坑。
- **每个新接入产品都进 onboarding 名单**（README 里列）。零接入产品 = 平台没有意义。
- **删除快于增加**：每季度回顾 platform-core 是否有可删的代码 / 配置 / 端点 / 抽象。
- **简化是负向工作**。没有"简化完了"那一天；只有"这周比上周少了多少认知负担"。

## 不做（明确）

- ❌ 在 identity_admin 加 K8s Secret 自动写入（2026-04-28 探索后否决——加层不简化）

- ❌ 多 cluster 同步 apps.yaml（业务还没复杂到需要）
- ❌ 完全自研 OIDC server 替代 Zitadel（Zitadel 是后台实现，对接入方隐藏即可）
