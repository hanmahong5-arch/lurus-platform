# 多租户简化设计（Lurus 平台）

> 触发：用户反馈"接登录还有什么多租户之类的，怎么就这么复杂呢"。
>
> 现状：account / org / org_member 三层结构对产品方完全暴露。
> 目标：默认对产品方零暴露 — 99% 产品只用 `account_id`，需要团队/组织功能的产品再单独 opt-in。

---

## 现状的复杂度（待治理）

接入 Lurus 的产品如果想做"邀请同事/共享余额/团队管理"，会被迫先理解：

1. **account**：单个用户。手机号/邮箱唯一。
2. **org**：组织。可以是个人独享（隐式 1 人 org）或多人协作。
3. **org_member**：account 在哪些 org 里以什么角色（owner / admin / member）。
4. **org_api_key**：组织级 API Key（与 account 个人 key 区分）。
5. **wallet**：钱包既能挂在 account 也能挂在 org。
6. **active org context**：用户登录后默认在哪个 org，谁能切换。

任何接入方光是搞清楚这六层关系就要半小时。绝大多数小工具型产品根本不需要。

---

## 简化原则

1. **默认值合理化**：account 不显式加入 org → 平台内部隐式给个 1 人 org（id 与 account_id 1:1 映射，对外不暴露）。
2. **产品 API 默认只返 account_id**。`org_id` 字段在 `/whoami` 等接口里默认隐藏。
3. **Org 是 opt-in 概念**：产品想用团队功能，显式调 `/api/v1/orgs/...` 系列。不调就当不存在。
4. **Wallet 默认绑 account**。"团队钱包"是 org 级 wallet，需要产品显式申请。
5. **API Key 默认是 account-scoped**。OrgAPIKey 是高级功能，单独申请。

---

## 对产品方的暴露面（新模型）

### Tier 0 — 工具型产品（默认）
**只需理解**：`account_id`（int64）。

```
GET /api/v1/whoami
{
  "account_id": 12345,
  "phone": "+861380...",  // 脱敏
  "email": "u@x.com",
  "plan": "pro"
}
```

完。没有 org、没有 member、没有角色。

### Tier 1 — 协作型产品（显式 opt-in）
产品在 dashboard 勾选"启用组织功能"后，产品的 INTERNAL_API_KEY 获得 `org` scope：

```
GET /api/v1/account/me/orgs
[
  {"org_id": 7, "name": "我的工作室", "role": "owner"},
  {"org_id": 12, "name": "海外项目组", "role": "member"}
]

POST /api/v1/orgs/12/active        # 切换默认 org
GET  /api/v1/orgs/12/members
GET  /api/v1/orgs/12/wallet
```

未启用时，这些端点对该产品的 INTERNAL_API_KEY 返回 403。

### Tier 2 — 企业版产品（白名单）
真正需要 RBAC、审计、SSO 同步的，单独商务对接，不暴露给一般接入方。

## 数据库不动，行为变化

不需要 schema 迁移。简化只发生在：

| 变化点 | 改法 |
|--------|------|
| `/api/v1/whoami` 默认响应 | 隐藏 `org_id` / `org_role` 字段，除非请求带 `?include=org` |
| `/api/v1/account/me` | 同上 |
| `INTERNAL_API_KEY` 创建 | 默认 scope 不含 `org:*`，需要时单独勾选 |
| Wallet 接口 | 默认 `wallet/balance` 查 account wallet；查 org wallet 走 `/orgs/:id/wallet` |
| 文档 | "接入指南"主线只讲 account_id，org 章节移到"高级"附录 |

---

## 落地步骤

1. **改 `/whoami` 响应过滤**（10 行 Go）
2. **改 INTERNAL_API_KEY 默认 scope**（DB 默认值改一处）
3. **更新 `接入指南.md`**，把 org 章节移到附录
4. **新接入方 onboarding 流程**：默认走 Tier 0，需要时手动升级

## 现有接入方影响评估

| 产品 | 是否依赖 org | 影响 |
|------|------------|------|
| Lucrum | 否（个人投资工具） | 无影响 |
| Tally | 是（进销存团队协作） | 需保留 org scope，已是 Tier 1 |
| Memorus | 否 | 无影响 |
| Forge | 视后续而定 | 默认 Tier 0 |
| 路途 APP | 否 | 无影响 |
| 2c-bs-www / Lucrum 公开站 | 否 | 无影响 |

**4 个产品是 Tier 0，1 个是 Tier 1**。简化对绝大多数产品是单向收益（少看一层抽象）。

## 心智模型对比

### 旧模型（产品方需要懂）
```
Account ─┬─ 我的钱包
         ├─ 我属于哪些 Org
         │   └─ Org 钱包（如果开了团队版）
         │       └─ 这次操作是扣 Org 还是扣 Account？
         └─ 我现在的 active org 是什么？
```

### 新模型（默认情况，产品方需要懂）
```
Account ── 钱包
```

如果产品有团队功能，再：
```
Account ── 我属于哪些 Org（显式 GET 才看得到）
```

---

## 与"两行接入"的关系

这是"两行接入"承诺的隐性前置：如果产品调 `/whoami` 还得处理 org 字段，就不是"两行"了。本简化保证：

```js
const user = await lurus.whoami()
// user = { account_id, email, plan }    ← 不要看到 org
```
